# TodoApi HTTP contract

Reference for the frontend team. Describes every endpoint the API currently
exposes, the JSON shapes on the wire, and the `ProblemDetails` error format
used for every failure response.

> **What this documents.** Whatever the controllers in `TodoApi/Controllers/`
> actually implement today — method, path, request/response bodies, status
> codes, and error shapes. Where the implementation diverges from the
> expected public contract (the `crunchloop/interview-tests` Postman
> collection), the drift is flagged in [Contract drift](#contract-drift) at
> the bottom — **not silently smoothed over in the endpoint bodies**.

## Conventions

### Base URL and media type

- All paths are rooted under `/api`.
- Success responses use `application/json; charset=utf-8`.
- Error responses use `application/problem+json; charset=utf-8`
  (RFC 7807). This is the **only** error shape the API emits — there are no
  plain-text bodies, no `{ "error": "…" }` envelopes.

### JSON casing

- Request and response bodies use **camelCase** property names (ASP.NET Core
  default). Examples below follow this.
- Identifiers:
  - TodoList `id` — **UUID v7** (`Guid`) generated by the API.
  - TodoItem `id` — **signed 64-bit integer** (`long`), database-assigned.

### Dates

- All timestamps are UTC, ISO-8601, e.g. `"2026-04-20T14:03:11.0000000Z"`.

### `ProblemDetails` shape (RFC 7807)

Every non-2xx response is a `ProblemDetails` / `ValidationProblemDetails`
object with the following fields:

| Field      | Type              | Notes |
|------------|-------------------|-------|
| `type`     | string (URI)      | Stable reference URL for the error family (may not resolve). |
| `title`    | string            | Short human-readable summary. |
| `status`   | integer           | HTTP status code (same as the response status). |
| `detail`   | string            | Longer human-readable description (= `Error.Description`). |
| `instance` | string            | Request path (added automatically by the framework). |
| `code`     | string (extension)| Machine-readable error code — stable, safe to branch on. |
| `traceId`  | string (extension)| Distributed-trace id, matches the `traceId` in server logs. |
| `errors`   | object (extension)| **Validation only.** Dictionary of field → array of messages. |

### Error.Definition → HTTP status map

The server's `Error.Definition` enum maps 1:1 to an HTTP status and a
`title` / `type`:

| `Error.Definition` | HTTP status | `title`                  |
|--------------------|-------------|--------------------------|
| `Validation`       | **400**     | `Validation Error`       |
| `Unauthorized`     | 401         | `Unauthorized`           |
| `Forbidden`        | 403         | *(not currently emitted)*|
| `NotFound`         | **404**     | `Resource Not Found`     |
| `Conflict`         | **409**     | `Conflict`               |
| `Error` / other    | **500**     | `Internal Server Error`  |

Frontends should branch on `status` first, and on the `code` extension
when finer granularity is needed.

### Validation error shape

```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "Validation Error",
  "status": 400,
  "detail": "Resource 'CreateTodoListCommand' has 1 validation error(s).",
  "instance": "/api/todolists",
  "code": "Validation",
  "traceId": "00-7c8b2f7e4b1e4f6aaf89c3b1e8b0dcae-1b3e2e4c4f5e7a1d-01",
  "errors": {
    "Name": ["'Name' must not be empty."]
  }
}
```

Notes:

- `errors` keys are the **command property names** (PascalCase, e.g.
  `Name`, `Id`, `TodoListId`, `ItemId`). They are **not** request-body
  field names. When the request body uses `name`, the corresponding
  validation key is `Name`.
- FluentValidation rules attach machine codes via `WithErrorCode(...)` (e.g.
  `todolists.name.empty`, `todolists.name.too_long`,
  `todoitems.listid.empty`, `todoitems.id.invalid`). These codes are **not
  currently surfaced** in the response body — only the error messages are.
  The top-level `code` extension is always the string `"Validation"` for
  validation failures, regardless of which rule fired.

### Not-found error shape

```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.5",
  "title": "Resource Not Found",
  "status": 404,
  "detail": "Resource 'TodoList' with identifier 'a4c1e2f8-5b2a-4e7c-9d3a-2f1e8b0dcae3' was not found.",
  "instance": "/api/todolists/a4c1e2f8-5b2a-4e7c-9d3a-2f1e8b0dcae3",
  "code": "NotFound",
  "traceId": "00-7c8b2f7e4b1e4f6aaf89c3b1e8b0dcae-1b3e2e4c4f5e7a1d-01"
}
```

### Conflict error shape

```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.10",
  "title": "Conflict",
  "status": 409,
  "detail": "<conflict-specific message>",
  "instance": "/api/todolists/…",
  "code": "Conflict",
  "traceId": "00-7c8b2f7e4b1e4f6aaf89c3b1e8b0dcae-1b3e2e4c4f5e7a1d-01"
}
```

> No endpoint today returns `Conflict` from its handler. It is listed so
> the frontend can wire up a generic handler for future cases (e.g.
> optimistic-concurrency failures).

### Unhandled 500

Any uncaught exception surfaces as a generic 500 `ProblemDetails`:

```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.6.1",
  "title": "Internal Server Error",
  "status": 500,
  "detail": "<generic message — never stack traces or internals>",
  "instance": "/api/…",
  "code": "Error",
  "traceId": "00-7c8b2f7e4b1e4f6aaf89c3b1e8b0dcae-1b3e2e4c4f5e7a1d-01"
}
```

Use `traceId` to correlate with server-side logs when reporting a 500.

## Contract drift

The public contract of record is the Postman collection under
[`crunchloop/interview-tests`](https://github.com/crunchloop/interview-tests)
(`integration/todo_list_api.postman_collection.json`). The endpoint docs
above describe the **actual controller behavior**, which diverges from that
contract in the ways listed here. These must be resolved (in code, or by
renegotiating the contract) before any frontend integration is considered
finished — do **not** build the frontend against the broken shapes below
without planning to follow up.

### 1. TodoItem resource segment — `items` vs `todos`

- **Controller (current):** `/api/todolists/{listId}/items`,
  `/api/todolists/{listId}/items/{itemId}`.
- **Contract expects:** `/api/todolists/{listId}/todos`,
  `/api/todolists/{listId}/todos/{itemId}`.

Breaks **every** item-level call in the Postman suite. Fix:
rename the route prefix on `TodoItemsCommandController` and
`TodoItemsQueryController` from `items` to `todos`.

### 2. Create endpoints return 201 Created, contract expects 200 OK

- **Controller (current):**
  - `POST /api/todolists` → `201 Created` + `Location` header.
  - `POST /api/todolists/{listId}/items` → `201 Created` + `Location` header.
- **Contract expects:** `200 OK` on both, no `Location` asserted.

The Postman tests assert `pm.response.to.have.status(200)` on create. A 201
response will fail the integration suite.

### 3. TodoItem request body — field names

- **Controller (current):** `{ "name": string }` on both POST and PUT to
  `/items` and `/items/{itemId}`.
- **Contract expects:**
  - POST body: `{ "description": string, "completed": boolean }`.
  - PUT body: `{ "description": string, "completed": boolean }`.

Missing `completed` on both; `name` vs `description` field rename.

### 4. TodoItem create response — no `id` field

- **Controller (current):** `AddTodoItemResponse` is
  `{ "todoListId": Guid, "name": string, "isComplete": bool }`. **No
  `id` is returned.** The Postman test then fails because it expects to
  read `jsonData.id` to use in later calls.
- **Contract expects:** JSON body containing `id` (plus at least
  `description` / `completed` echoes).

This one is the worst of the drifts — even renaming the route won't fix it;
the handler has to be changed to return the created item's numeric id in
the response body, and the response record needs `Id`, `Description`,
`Completed` fields instead of the current `TodoListId`/`Name`/`IsComplete`.

### 5. TodoItem update response — field names

- **Controller (current):** `{ "id", "todoListId", "name", "isComplete" }`.
- **Contract expects:** response that echoes `description` and `completed`
  (plus `id`).

Same rename as (3) + (4) applied to `UpdateTodoItemResponse`.

### 6. `PUT .../complete` endpoint is extra

- **Controller (current):** `PUT /api/todolists/{listId}/items/{itemId}/complete`
  toggles `IsComplete = true`.
- **Contract expects:** completion is expressed by the regular
  update call with `"completed": true` in the body. There is no
  `/complete` sub-path in the Postman suite.

The extra endpoint is not tested, so it does not break the suite, but it
is dead weight for the frontend — the frontend only needs the single
`PUT /todos/{itemId}` with `completed` flag. Either drop
`CompleteTodoItem` or repurpose it internally; don't expose both paths.

### 7. TodoList create/update response shape — unasserted but inconsistent

- **Controller (current):**
  - `POST /api/todolists` → `CreateTodoListResponse = { id, name, createdAt }`.
  - `PUT /api/todolists/{id}` → `UpdateTodoListResponse = { id, name, updatedAt }`.
- **Contract expects:** Postman only checks for `id` on POST and for the
  new `name` on PUT. Both current shapes satisfy that, so this is
  **not a failing drift** — but note the `createdAt` / `updatedAt`
  asymmetry (create exposes creation time, update exposes update time but
  drops creation time). If you care about consistency, either include
  both timestamps on both responses or return just `{ id, name }`.

### Summary

| #  | Area                               | Breaks Postman? |
|----|------------------------------------|-----------------|
| 1  | Item route segment `items`/`todos` | **Yes**         |
| 2  | POST returns 201 instead of 200    | **Yes**         |
| 3  | Item body `name` vs `description`+`completed` | **Yes** |
| 4  | Item create response missing `id`  | **Yes**         |
| 5  | Item update response field names   | **Yes**         |
| 6  | Extra `/complete` endpoint         | No (just bloat) |
| 7  | List response shape asymmetry      | No              |

Frontend should treat the endpoint docs as *current-as-of-build-hash*;
expect (1)–(5) to change before this API is considered conformant.
