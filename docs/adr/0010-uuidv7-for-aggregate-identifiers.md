# 0010 - Use UUIDv7 (Guid v7) for aggregate identifiers

- Status: accepted
- Date: 2026-04-20
- Deciders: TodoApi team

## Context and problem statement

ADR-0003 pins the DDD building blocks — entities have identity,
aggregates have a root, repositories work in terms of the root id —
but it deliberately does not name the identifier *type*. The starter
template uses `long` (database-assigned `IDENTITY` columns) on
`TodoList.Id` and `TodoItem.Id`. As we rewrite modules we have to pick
a deliberate identifier type; picking it once, now, avoids every
module re-deciding.

The repo already ships a hand-rolled `GuidV7.NewGuid()` helper at
`TodoApi/Infrastructure/Extensions/GuidV7.cs`, which indicates the
previous direction of travel. This ADR makes that direction explicit
and writes down the trade-offs.

## Decision drivers

- **Sortability** — ids generated close together should sort close
  together so that inserts cluster well on a primary index (SQL
  Server defaults to clustered PKs).
- **Externalizability** — ids appear in URLs and JSON bodies. We want
  them opaque enough to not leak row counts or ordering hints the way
  sequential `long`s do, but still safe to log and share.
- **No round-trip to the DB to know an id** — domain code and tests
  should be able to construct a new aggregate and know its id before
  `SaveChangesAsync`, so that commands can return the id in the HTTP
  response without an extra query.
- **No cross-module coupling through a shared sequence** — per
  ADR-0005 each module has its own `DbContext`; an id strategy that
  requires a central sequence generator (like a Hi/Lo table) would
  contradict module isolation.
- **Future sync** — when sync arrives (explicitly deferred, per
  CLAUDE.md), ids will flow to an external system. Globally unique
  ids make that handoff trivial; `long` ids would require a mapping
  table.
- **Cost on .NET 8 today, near-zero on .NET 9+** — the BCL ships
  `Guid.CreateVersion7()` starting in .NET 9; once we bump TFM this
  becomes a one-liner and the helper goes away.

## Considered options

- **UUIDv7 (Guid v7).** 48-bit Unix-ms timestamp prefix + 74 bits
  random + version/variant bits. Time-ordered, globally unique,
  widely supported (RFC 9562, 2024).
- **UUIDv4 (plain `Guid.NewGuid()`).** Globally unique but fully
  random — terrible for clustered indexes (page splits on insert).
- **`long` with database `IDENTITY`.** Cheap, small, sortable, but
  requires a DB round-trip for the id, leaks creation order, and
  complicates cross-module / external-system integration.
- **ULID / NanoID.** Similar sorting properties to UUIDv7 but string-
  shaped and less well supported by EF Core out of the box.
- **Sequential GUIDs (`NEWSEQUENTIALID()`).** SQL Server-specific;
  only sortable inside one machine; ties the API to SQL Server for
  id semantics.

## Decision outcome

Chosen option: **UUIDv7 stored as `System.Guid`**, generated by
`GuidV7.NewGuid()` until the TFM is bumped to net9.0+, at which
point the helper is replaced by `Guid.CreateVersion7()` in the BCL.

Conventions:

- Every aggregate root has a `Guid Id` property. No `long`
  identifiers on new aggregates.
- Ids are assigned in the aggregate's constructor via
  `GuidV7.NewGuid()`. The domain does not delegate id generation to
  the persistence layer — the aggregate is internally consistent the
  moment it is constructed.
- EF Core maps the id as `uniqueidentifier` on SQL Server. Do **not**
  configure `.ValueGeneratedOnAdd()` for it — the app is the source
  of truth for the id.
- Clustered index: the default clustered PK on the id is acceptable
  because v7 ids are approximately time-ordered. If a hot table later
  shows insert-side hot-spotting, revisit with a separate ADR (e.g.
  `NEWSEQUENTIALID` default, non-clustered PK).
- Public API: ids appear as their canonical 36-character textual form
  in URLs and JSON. Route templates use `{id:guid}` for model
  binding.
- On net9.0+: delete `Infrastructure/Extensions/GuidV7.cs`, replace
  call sites with `Guid.CreateVersion7()`. No ADR needed for that
  swap — it is a mechanical move documented by the TODOs in the
  helper.
- Monotonicity within a millisecond is **not guaranteed** by the
  current helper (see TODO in `GuidV7.cs` referencing RFC 9562 §6.2).
  If strict ordering is needed for pagination cursors, that is a
  future ADR.

### Consequences

- Positive: inserts cluster well on a time-ordered key; no page-split
  storm from random v4 Guids.
- Positive: domain objects are complete the moment they are
  constructed — tests and handlers do not have to pretend the DB
  already saved the row.
- Positive: ids are globally unique, so future sync work can keep the
  same id across systems without a mapping table.
- Positive: no central sequence generator, so module isolation
  (ADR-0005) is preserved.
- Negative: 16 bytes per id instead of 8 for a `long`. Acceptable
  trade for the properties above at this scale.
- Negative: **breaking change to the public API contract**. Existing
  endpoints expose `long` ids; moving to `Guid` changes resource URLs
  and payload shapes. The integration tests at
  <https://github.com/crunchloop/interview-tests> must be re-read and
  confirmed to accept the new id format before the first module is
  switched over. If they don't, we either (a) coordinate an update to
  those tests, or (b) supersede this ADR for the TodoLists module
  specifically. This is a gating concern, not a blocker.
- Negative: migration of existing data (current `long`-keyed rows)
  requires either a one-way cutover or a parallel-write period. We
  defer that detail to the first module's migration PR — ids on green
  modules are `Guid` from day one, and the starter tables get
  rebuilt as part of the rewrite.
- Neutral: on net9.0+ the helper becomes redundant and is removed.
